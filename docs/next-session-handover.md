# Fire Forex — next-session handover

> Paste this whole file into a fresh Claude Code session to pick up work.

## What this is

Fire Forex is a **local forex-strategy optimisation system**. A FastAPI
backend + single-page HTML/JS/Tailwind frontend drive a Rust backtest
engine (`ff_core`, built locally from core/ via maturin) over Dukascopy parquet
data on `G:\My Drive\BackTestData`. The user is non-technical and
controls everything through the web UI — no Python editing required.

## Starting it

```powershell
cd "C:\Users\ROG\Projects\Fire Forex"
.\.venv\Scripts\python.exe -m pip install -r requirements-web.txt  # first time only (uv under the hood)
.\.venv\Scripts\python.exe run.py web
```

Open **http://127.0.0.1:8000/** — one-page app with four tabs.

## The four tabs

- **Parameters.** Pick pair / main TF / sub TF. The **Complexity** slider
  (1–10) is a convenience preset for step granularity + which features
  are on. **Step granularity** and **Feature set** buttons give finer
  control; the per-feature tickboxes always win over both. Every feature
  (stop loss, take profit, trailing, breakeven, partial, stale, session,
  max-bars, days) is listed MT5-style with a checkbox + **Min · Step ·
  Max** columns. Signal families (EMA / MACD / Donchian) have their own
  master tickboxes.

- **Run.** Trials (10–50,000), seed, optional run label, one big **Run
  backtest** button. One run at a time. A heartbeat keeps the progress
  bar moving during the Rust sweep (it used to stall at 45%).

- **Results.** Eight KPI tiles with a delta-vs-baseline line when a
  baseline is pinned, orange equity curve, winning parameters rendered in
  plain English, **Pin as baseline** / **Clear baseline** buttons, and a
  link to the multi-run comparison dashboard.

- **History.** Last 30 rows of `artifacts/history.csv`. Every row has a
  **Pin** button — any past run can become the baseline.

Every `?` pill opens a tooltip sourced from
`docs/knob-explanations.md`.

## The three UI concepts

1. **Complexity slider** — one-click preset. Low = minimal features,
   coarse steps. High = all features on, fine steps. Moving it wipes
   per-knob overrides.
2. **Step granularity** (Coarse/Normal/Fine/Finest) — multiplies every
   numeric step. Moving it wipes per-knob step overrides only.
3. **Feature set** (Minimal/Balanced/All-on) — bulk on/off for the
   optional exit features AND the signal families (dropping families is
   the main way to cut runtime).

## The mental model

An EA is four things:

- `data`       — pair + main_tf + sub_tf (fills).
- `execution`  — pip value, commission, spread cap, ATR period.
- `signals`    — one or more entry-signal families with integer grids.
- `engine_schema` — the tunable parameter tree (Branches / Groups / leaves
  from `ff/schema.py`).
- `engine_mapping` — maps the schema onto the 27 Rust slot columns. Not
  user-editable; generated by `ff/defaults/complexity.py`.

User overrides are a separate dict layered on top at request time:

```json
{
  "groups":          {"trailing": true, "partial": false},
  "signal_families": {"ema_cross": true, "donchian": false},
  "knobs":           {"stop_loss.atr.mult": {"min": 1.0, "max": 3.0, "step": 0.1}},
  "global":          {"step_multiplier": 1.0}
}
```

Applied in `ff/defaults/overrides.py::apply_overrides()`.

## Baselines

Pinning a run writes `artifacts/baseline.json`. Future runs show delta
tiles on the Results tab. Pin from the Results tab (current run) or from
any row in the History tab (past run, via the Pin button).

## Known rough edges

- `pair_tf.yaml` is a legacy fallback. Real ranges come from
  `ff/defaults/volatility.py::derive_ranges()` using median 14-bar ATR
  cached at `artifacts/volatility_cache.json`. Delete the cache to
  force a rescan.
- Only `random` optimiser today. Optuna is the next step.
- Progress percentages are interpolated during the Rust sweep — they're
  *indicative*, not exact.

## What NOT to do

- Don't fork the Rust backtester — it's a wheel, feature additions go in
  `docs/rust-wishlist.md`.
- Don't reintroduce Streamlit, Gradio, or any Python-rendered UI.
- Don't bind the server to anything other than `127.0.0.1`.
- Don't touch `eas/complex01.py`'s `ENGINE_MAPPING` — regenerating from
  JSON is the supported path.

## Data tab runbook — "the UI says EURUSD has no data"

Symptoms and the right first move:

- **Inventory table empty / health modal spins forever.** Almost always
  stale uvicorn processes holding :8000 with an older route table.
  Run `.\scripts\ff_kill_server.ps1` (kills every python running
  `run.py web` + any stray :8000 listener), then `.\scripts\ff_start_server.ps1`
  to relaunch in the foreground. Keep that terminal in view — closing
  it terminates the server cleanly. Repeatedly backgrounding the
  server is what creates the zombie PIDs in the first place.
- **Health modal shows "scanning parquet… cold files on Google Drive
  can take 1–2 min."** That's normal. Cold reads of the ~93 MB M1
  files from `G:\My Drive\BackTestData\` legitimately take 60–120 s
  on the first touch. The modal has a 120 s client-side timeout; if
  it hits, retry once — the second read comes back in <1 s from the
  warm page cache. If the second read also times out, the server is
  hung: kill + restart.
- **"no data for … in this run" on the Results-tab scatter chart** is
  *not* a disk/data problem. It means the selected metric column was
  all-null across every trial of the run — e.g. selecting DSR on a
  run with zero trades. Switch the metric dropdown or pick a run that
  actually produced trades.
- **Filenames on disk use `EUR_USD_M1.parquet` with an underscore.**
  Every code path (inventory regex, harness loader, frontend select
  values) agrees on that. If you ever see a `EURUSD` without the
  underscore, it's the scatter-chart empty-state copy — not a file
  mismatch.
